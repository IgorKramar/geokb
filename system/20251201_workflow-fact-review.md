# Рабочий процесс факт-ревью базы знаний по географии

Описание процесса проверки существующих заметок на фактические, смысловые и логические несоответствия, несостыковки и ошибки.

## Основной принцип

При факт-ревью ИИ должен:
1. **Анализировать существующие заметки** — систематически проверять заметки на консистентность
2. **Выявлять несоответствия** — находить фактические, смысловые и логические ошибки
3. **Составлять репорт** — создавать информативный документ с найденными проблемами
4. **Предлагать решения** — формулировать предложения по улучшению и задавать уточняющие вопросы

## Типы проверяемых проблем

### 1. Фактические несоответствия

- **Противоречивые факты** — одна заметка утверждает одно, другая — другое
- **Неточные географические данные** — несоответствия в описании стран, городов, регионов
- **Неправильные названия** — разные варианты написания одного и того же
- **Ошибочные связи** — неправильные связи между концепциями
- **Устаревшая информация** — информация, которая была изменена в других заметках

**Примеры:**
- Заметка A: "Дорожные знаки Германии имеют синий фон"
- Заметка B: "Дорожные знаки Германии имеют белый фон"
- → Противоречие в описании дорожных знаков

### 2. Смысловые несоответствия

- **Противоречивые описания** — разные версии одного и того же признака/факта
- **Несогласованные концепции** — концепции, которые не согласуются друг с другом
- **Логические разрывы** — отсутствие связи между связанными концепциями
- **Дублирование информации** — одна и та же информация в разных заметках с разными формулировками

**Примеры:**
- Заметка A: "Франция — страна с умеренным климатом"
- Заметка B: "Франция — страна с субтропическим климатом"
- → Противоречие в описании климата (возможно, речь о разных регионах)

### 3. Логические ошибки

- **Географические несоответствия** — локации, которые не могут существовать одновременно
- **Климатические несоответствия** — описания климата, которые противоречат географическому положению
- **Растительные несоответствия** — описания растительности, которые не соответствуют климату
- **Архитектурные несоответствия** — описания архитектуры, которые не соответствуют региону

**Примеры:**
- Страна находится в тропиках, но описана как страна с хвойными лесами
- → Логическая ошибка: хвойные леса не характерны для тропиков

### 4. Структурные проблемы

- **Отсутствующие связи** — заметки, которые должны быть связаны, но не связаны
- **Неправильные ссылки** — ссылки на несуществующие или неправильные файлы
- **Неполная информация** — заметки, которые должны содержать больше информации
- **Избыточная информация** — информация, которая дублируется в нескольких заметках

**Примеры:**
- Заметка о стране упоминает город, но нет ссылки на заметку о городе
- Заметка о признаке не связана с заметками о странах, где он встречается

### 5. Консистентность терминологии

- **Разные названия одного и того же** — один концепт называется по-разному в разных заметках
- **Неправильные склонения** — неправильное использование падежей в ссылках
- **Несогласованная транслитерация** — разные варианты написания имен собственных

**Примеры:**
- "США" в одной заметке, "United States" в другой
- "Москва" vs. "Moscow" (разные варианты написания)

## Пошаговый процесс факт-ревью

### 1. Подготовка к ревью

#### 1.1. Определение области проверки

ИИ определяет, что проверять:
- **Тематическая область** — конкретная тема (например, "дорожные знаки", "архитектура", "растительность")
- **Географическая область** — конкретный регион или страна
- **Тип заметок** — страны, города, признаки
- **Полное ревью** — проверка всех заметок по географии

#### 1.2. Сбор заметок для проверки

- Использую MOC-заметки для навигации
- Ищу связанные заметки через семантический поиск
- Собираю все заметки, относящиеся к выбранной области

### 2. Систематический анализ заметок

#### 2.1. Чтение и анализ каждой заметки

Для каждой заметки проверяю:
- **Содержание** — что утверждается в заметке
- **Ссылки** — на какие заметки ссылается
- **Контекст** — как заметка вписывается в общую картину
- **Факты** — какие конкретные факты утверждаются

#### 2.2. Сравнение с другими заметками

Для каждого факта проверяю:
- **Есть ли другие заметки**, которые утверждают то же самое?
- **Есть ли противоречия** с другими заметками?
- **Согласуются ли связи** между заметками?
- **Логична ли последовательность** фактов?

#### 2.3. Проверка связей

Для каждой связи проверяю:
- **Существует ли связанная заметка?**
- **Правильна ли ссылка?** (правильное имя файла)
- **Двусторонняя ли связь?** (если заметка A ссылается на B, должна ли B ссылаться на A?)
- **Логична ли связь?** (имеет ли смысл эта связь?)

### 3. Выявление проблем

#### 3.1. Категоризация проблем

Каждую найденную проблему категоризирую:
- **Тип проблемы** (фактическая, смысловая, логическая, структурная, терминологическая)
- **Серьезность** (критическая, важная, незначительная)
- **Затронутые заметки** (какие заметки связаны с проблемой)
- **Контекст** (в каком контексте возникает проблема)

#### 3.2. Документирование проблем

Для каждой проблемы записываю:
- **Описание проблемы** — что именно не так
- **Затронутые заметки** — какие заметки содержат проблему
- **Контекст** — где и как проявляется проблема
- **Примеры** — конкретные цитаты или ссылки на проблемные места

### 4. Формирование вопросов

#### 4.1. Уточняющие вопросы

Для каждой проблемы формирую вопросы:
- **Вопросы для уточнения фактов** — что является правильным фактом?
- **Вопросы для уточнения смысла** — какая версия является правильной?
- **Вопросы для уточнения логики** — как должна быть выстроена последовательность?
- **Вопросы для уточнения связей** — как должны быть связаны концепции?

#### 4.2. Принцип максимума вопросов

Задаю **максимум вопросов**, которые могут помочь:
- Не ограничиваюсь минимальным набором
- Задаю уточняющие вопросы
- Предлагаю варианты решения, если возможно
- Спрашиваю о деталях, которые могут быть важны

**Пример вопросов для проблемы:**
```
Проблема: Противоречие в описании дорожных знаков Германии
- Заметка A: "Дорожные знаки Германии имеют синий фон"
- Заметка B: "Дорожные знаки Германии имеют белый фон"

Вопросы:
1. Какой цвет фона у дорожных знаков Германии на самом деле?
2. Может ли быть, что в разных заметках речь идет о разных типах знаков?
3. Есть ли региональные различия в дорожных знаках Германии?
4. Нужно ли обновить одну из заметок или обе?
```

### 5. Предложения по улучшению

#### 5.1. Конкретные предложения

Для каждой проблемы предлагаю:
- **Конкретное решение** — что именно нужно изменить
- **Способ исправления** — как исправить проблему
- **Приоритет** — насколько важно исправить эту проблему
- **Зависимости** — какие другие изменения могут потребоваться

#### 5.2. Структура предложения

Каждое предложение содержит:
- **Что изменить** — конкретное изменение
- **Где изменить** — в каких заметках
- **Как изменить** — конкретные шаги
- **Почему** — обоснование необходимости изменения

**Пример предложения:**
```
Предложение: Унифицировать описание дорожных знаков Германии

Что изменить: Обновить описание дорожных знаков в заметке B с "белый фон" на "синий фон"

Где изменить:
- src/20250101_germany.md (если существует)
- src/20250101_road-signs-europe.md (где упоминается Германия)

Как изменить:
1. Проверить правильное описание с пользователем
2. Обновить все упоминания дорожных знаков Германии
3. Проверить, что нет других противоречий

Почему: Консистентность фактов важна для достоверности базы знаний
```

### 6. Создание репорта

#### 6.1. Структура репорта

Репорт создается в папке `system/` с именем:
- Формат: `YYYYMMDD_fact-review-[тема].md`
- Пример: `20251201_fact-review-european-signs.md`
- Для полного ревью: `YYYYMMDD_fact-review-full.md`

#### 6.2. Содержание репорта

Репорт содержит:

**1. Метаданные**
- Дата ревью
- Область проверки
- Количество проверенных заметок
- Количество найденных проблем

**2. Резюме**
- Общая статистика проблем
- Распределение по типам проблем
- Приоритетные проблемы

**3. Детальный список проблем**

Для каждой проблемы:
- **Название проблемы**
- **Тип** (фактическая, смысловая, логическая и т.д.)
- **Серьезность** (критическая, важная, незначительная)
- **Описание** — что именно не так
- **Затронутые заметки** — список заметок с проблемой
- **Контекст** — где и как проявляется
- **Примеры** — конкретные цитаты или ссылки

**4. Вопросы для уточнения**

Список всех вопросов, сгруппированных по проблемам:
- Вопросы для каждой проблемы
- Варианты ответов, если возможно
- Контекст вопросов

**5. Предложения по улучшению**

Список всех предложений, сгруппированных по проблемам:
- Конкретные предложения для каждой проблемы
- Приоритет исправления
- Зависимости между исправлениями

**6. План действий**

Рекомендуемый порядок исправления проблем:
- Какие проблемы исправить в первую очередь
- Какие проблемы можно исправить вместе
- Какие проблемы требуют дополнительной информации

#### 6.3. Формат репорта

Репорт оформляется в Markdown с использованием:
- Заголовков для структуры
- Таблиц для статистики
- Списков для проблем, вопросов и предложений
- Блоков кода для примеров
- Ссылок на заметки

### 7. Обновление после исправлений

#### 7.1. Отслеживание исправлений

После того, как пользователь исправит проблемы:
- Обновляю репорт, отмечая исправленные проблемы
- Проверяю, что исправления решают проблемы
- Выявляю новые проблемы, которые могли возникнуть

#### 7.2. Повторное ревью

При необходимости провожу повторное ревью:
- Проверяю исправленные заметки
- Убеждаюсь, что проблемы решены
- Проверяю, не появились ли новые проблемы

## Интеграция в рабочий процесс

### Периодическое ревью

Рекомендуется проводить факт-ревью:
- **После больших изменений** — после обработки большого объема информации
- **Перед важными обновлениями** — перед публикацией или использованием базы знаний
- **Регулярно** — раз в месяц или квартал для поддержания качества

### После обработки информации

После обработки входящей информации по географии:
1. Запускаю `python kb.py all` для обновления мета-файлов
2. При необходимости провожу факт-ревью новых или измененных заметок
3. Создаю репорт, если найдены проблемы

### Связь с другими процессами

- **Бедные заметки**: При наполнении бедных заметок проверяю консистентность новой информации
- **Битые ссылки**: При создании заметок по битым ссылкам проверяю консистентность с существующими заметками
- **Входящая информация**: При обработке новой информации проверяю на противоречия с существующими заметками

1. **Систематичность** — проверяю все заметки систематически, не пропуская
2. **Контекст** — всегда учитываю контекст при выявлении проблем
3. **Максимум вопросов** — задаю все вопросы, которые могут помочь решить проблему
4. **Конкретные предложения** — предлагаю конкретные, выполнимые решения
5. **Приоритизация** — определяю приоритет исправления проблем
6. **Документирование** — документирую все найденные проблемы с примерами
7. **Структурированность** — репорт структурирован и легко читается
